Petuum 入门

发表于 04/23/2017 | 分类于 大数据工具学习

有幸接到了来自Petuum Inc.的面试。
本来对其了解不多，google了相关资料后，对这个startup充满了期待。他们开发的Parallel Machine Learning System（PMLS）具有非凡的创造力和发展潜力。真心希望能有机会加入他们团队。
以下是关于Petuum的学习总结。

简介

Petuum是一个新兴的，基于大数据的分布式机器学习平台。其主要目的是为了解决当前分布式机器学习框架所面临的资源利用率低，通用性差问题。由于机器学习算法本身具有“容错性”，“动态结构依赖”和“非均匀收敛”等特点，Petuum采用称为Stale Synchronous Parallel（SSP，自译为“陈旧同步并行”）的一致性模型，来保证整个系统的资源高效，可扩展性，正确性和通用性。

目前市面上也存在一些分布式机器学习的工具或者平台，但各有其弊端。类似于TensorFlow和CNTK等机器学习工具虽然保证里计算的高效性和结果的正确性，但是其开发是针对于某一类特定的机器学习问题的，因此并不具备通用性；而更通用的大数据平台，例如Spark，其一致性模型是Bulk Synchronous Parallel(BSP，自译为“块同步并行”)。而Petuum的提出正是为了解决这个问题。

另注，Petuum将实现支持全数据源（Omni-source），全语言（Omni-Lingual），全平台（Omni-Mount）。不确定Petuum的开发计划是怎样的，如果有下轮面试希望可以问一下。

Bosen, Poseidon and Strads

Petuum将机器学习问题分为数据并行（Data Parallel）和模型并行（Model Parallel）两个子问题。基于这个假设，Petuum平台引入了三个概念，Scheduler，Parameter Server和Worker。

  • Scheduler: 通过允许用户控制Worker机器更行哪些模型参数来实现模型并行；
  • Worker: 接收Scheduler调度，并更行模型参数的节点，是真正执行计算的组件；
  • Parameter Server: 存储模型参数的地方，其使用了基于表的KV对的存储模式，并实现了SSP模型以保证参数一致。

在Petuum现有的PMLS上，Strads对应Scheduler，Bosen对应Workder和Parameter Server，而Poseidon是专门从Bosen中提取出来，用于解决Deep Learning问题的子框架。

Stale Synchronous Parallel

个人的理解是，SSP保证了Petuum的高效性和正确性，大大减少了Petuum平台的网络通信用时，是整个Petuum的核心，是Petuum能优于其他分布式机器学习框架的最主要原因。（当然，仅仅是个人理解，不具有科学性。）
在SSP模型下，快速的Worker会从本地缓存中读写陈旧版本的参数进行新的计算，而不是等待Central Storage传回最新的参数；之所以这种方法行得通，是由于机器学习模型本身具有的容错性的特点。特别是在多次迭代之后，参数的偏差会趋近于0。因而，SSP模型最终是可以保证收敛的。同时，我们可以通过限制stale阀值的大小，来控制偏差出错的可能性，保证结果正确性。

与SSP相对应的，还存在BSP和Asynchronous两种同步机制。

  • BSP: 所有Worker在迭代完成之后，必须进行高密度的参数同步通信，这就导致网络通信次数大大提高；而本身网络通信就是分布式系统性能提升的瓶颈，所以BSP的效率很低；同时，快速的Worker需要等待慢速的Worker完成计算才能接收到参数更新，也就浪费了大量的等待时间；
  • Asynchronous: Worker允许在任何时候以任何方式更新参数，这就导致速度不同的Worker获取的参数不一致，从而导致最终结果不收敛。

综上，SSP既能保证结果收敛，又能保证运算的高效性。当SSP的stale阀值为0，SSP=BSP；当SSP的stale阀值趋于无穷，SSP=Asynchronous。
