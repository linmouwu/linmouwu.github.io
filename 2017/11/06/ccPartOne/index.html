Cloud Computing Part 1

发表于 11/06/2017 | 分类于 Coursera

对于我这样的懒人而言，梳理知识点真真算不得一件快乐的事情。特别是当知识点杂乱无序，而需要我自己抽丝剥茧般理出个所以然时，更是让人头疼。上完了Coursera上Machine Learning的课程之后，选修了UIUC的Cloud Computing的课程。这是一个specilization，也就是由多门课组成。估计上完这个系列我就该累死了，哈哈哈哈哈。我一定是脑子哪里长洞了。

简而言之，这一系列课程的前两节，主要讲的是云计算的基础知识。虽然在CMU的时候上过CC的课，但是一来早就忘光了，二来两所学校的知识体系也不完全一样，所以还是学了不少新概念。当然这些概念能掌握多牢固能记得多久，就看自己的造化了。废话不多说，开始整理第一部分的内容把。

第一部分分5周，分别对应下文的每个section。因为这个课程知识基础知识入门，且因为我懒，所以目前都是在梳理理论知识而已。以后有空，想起来要翻翻看的时候，再回头来看吧。

Week One - Intro to Cloud

  • 云 ＝ 大量存储空间 ＋ 计算核心；
  • Cloud computing 的四个特征：Massive scale， On-demand process， Data-intensive Nature， Cloud computing programming paradigms；
  • WUE = Annual water usage/IT equipment energy
  • PUE = Total facility power/IT equipment power
  • Distributed system：是一系列实体的集合，每个实体都是autonomous, programmable, asynchronous and failure-prone， and communicate through an unreliable communication medium.
  • MapReduce：在map和reduce之间存在一步partition，使得对应的key分配到确定的reduce server，partition的方法一般是hash。
  • Hadoop MapReduce： 初始输入和最终输出都是写在分布式存储系统中，但是中间结果，例如Map的输出，Reducer的输入，均写在local disk of worker.
  • Hadoop YARN： yet another resource negotiator， 用于管理资源分配；包括Global Resource Manager(RM)，Per-server Node Manager(NM)，Per-application Master(AM).
  • YARN 容错：NM heartbeats to RM， NM tracking applications， AM heartbeats to RM；
  • Slow server： Perform duplicated execution，this is called speculative execution；

Week Two - Membership

  • MTTF：平均容错时间：A＊a = B＊b， A， B表示机器数量，a，b表示每一台机器无差错工作时间；
  • 三种Membership protocols： Complete list(virtual synchrony)， Almost-complete list(Gossip, SWIM)， Partial-random list(SCAMP)
  • 完备性completeness： 所有错误都被检测到；准确性accuracy：没有错误的检测；速度speed：从出现错误到检测到的时间；规模scale：每个member都有相近的work load；
  • centralized heartbeat可能出现单一节点fail；ring heartbeats没法防止多个节点同时fail；all-to-all才能有效保障completeness和容错性；
  • Gossip-style：定期向身边member发出信号更新时间戳；分为两个timeout，第一个timeout用来检测fail，第二个timeout用来delete from table：防止收到不正确的更新而使得数据无法删除；
  • 在gossip中，不选择所有的节点，而只选择部分节点进行gossip，这样可以将每个节点的工作量下降到logN；
  • SWIM：向随机K的过程发送请求，让K尝试范围pj，如果任意访问成功，则pj没有fail；时间复杂度也是O(logN)；
  • Tree-based multicast protocols：SRM(scalable reliable multicast) and RMTP(reliable multicast transport protocol)
  • Push：定期向周围成员（subset） 发送message；Pull：定期向周围成员（subset）获取message；
  • 前期未收到message的成员多，适合push；后期收到过message的成员多，适合pull；需要传输的数据量大致只有logN；而时间复杂度前期为logN，后期为loglogN；

Week Three - P2P

  • P2P：Peer-to-peer，分布式系统中常见的模式；
  • Naspter：分为server和client，server存储directory，包括文件名和文件基本信息；client链接到server，存储文件本身；查询过程如下：
  1. client发送query到server；
  2. 所有server查找diretory，然后返回对应的文件所在的host的ip和port＃；
  3. client ping到最合适的host，这个host也是另外的client，并下载文件；
  4. 所有沟通都是TCP；
  • Gnutella：只有client，也叫servents，servents存储文件本身，以及到邻近servents的指针；Gnutella包括5中不同的信息：
  1. query：查找；
  2. queryhit：查找成功回复；
  3. ping：探测其它邻接点；
  4. pong：回复探测；
  5. push：指示准备开始文件传输；通过requestor push一个信息建立TCP通道；
  • Gnutella为了避免海量数据交换，每个信息会有id和TTL，id用来标示已经收过的信息，TTL标示信息存活的时间长度；queryhit和pong都是沿原路返回；
  • FastTrack： 跟Gnutella类似，但是存在supernode，存储directory listing nearby files，supernode会不定期更换；
  • BitTorrent： 文件被分割成小块存储在不同的peer上，采取local rarest first策略下载blocks，一个block可能存在于多个peer上，选择最快的下载；BitTorrent的几个概念：
  1. Tracker：维护peer的信息；
  2. Peer：提供文件下载和上传；
  3. .torrent file：种子文件，存储tracker信息和文件信息；当peer下载.torrent所对应的文件时，自己成文seed，可以提供文件下载；
  • Chord： 环形分布式系统，通过哈希函数确定文件存储位置（取最近的下一个节点）；此算法的时间和空间复杂度均为logN；
  • 在chord中，当需要寻找文件时，从当前peer的最远端开始，逐步向文件的哈希函数值逼近；
  • Pastry： routing table based prefix matching；每个peer维护一个prefix matching table；
  • Kelips： node会被分成不同的group，不同group之间会有contact node；文件会在本group和其它group中都有副本；lookup时先查找本group，然后跨group；

    Week Four - NoSQL

  • NoSQL：not only SQL，unstructured， no schema imposed， column storage；
  • Cassandra： 运用chord进行文件存储，但是不直接使用finger table，而是哈希函数；运用snitch表示节点位置：x.dc.rack.node；
  • 分布式写有多种策略：包括timestamp， quorum， coordinator等；
  • 对于cassandra而言： 先写到local log，然后写到memtable，最后当memtable满了，写到SSTable with data file， index file and bloom filter；
  • bloom filter 通过多个哈希函数，标记每个k是否存储large bit map中；当所有哈希值均有对应，则key存在；
  • Cassandra中任何一个node都可以是coordinator，所以需要维护所有的存活node的列表；其membership维护方式是gossip；
  • Consistency：strong consistency，casual consistency， eventual consistency；
  • HBase：结构大致为：HRegionServer - HRegion - Store - MemStore and StoreFile - HFile；
  • HRegionServer是server本身，其下的region包括了replicated 的数据；
  • clock skew： 时间数值上的差别；非0说明时间不同步；
  • clock drift： 时间速度上的差别；非0导致skew非0；
  • Maximum drift rate：最大时间差比率， 当skew为M，我们需要不断每M/2*MDR的时间就同步一次；
  • External sync和internal sync：表示当前时钟对外和对group内部时钟的同步；当组外同步差值最大为D，则组内同步差值最大可能为2＊D；
  • NTP： network time protocal；
  • Lamport timestamp：每个进程初始时间为0，当发生动作或者发送信息时，时间戳＋1；当接到信息时，时间戳为当前值和消息时间戳的值中较大者＋1；
  • Vector clock：与时间戳类似，只是维护的是vector而非单一timestamp；通过不同的vector的数值比较，可以判断事件之间是否可以并发；Vi <= Vj 则不可；
